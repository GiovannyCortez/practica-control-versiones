# practica-control-versiones


## Descripción de  lo realizado en clase 
implementación de una práctica de control de versiones, utilizando la informacion proporcinada  en Guía para que apliquen los conceptos del video en GitHub

---

## Estructura del repository

- **main**  Rama principal (Trunk) 
- **feature-login** Rama de desarrollo (Branch)
- **v1.0** Versión final del sistema (Tag) 

---

## Flujo de trabajo aplicado

Lo que realice durante la practica paso a paso fue lo siguiente:

1. Clonación del repositorio  a mi dispositivo localmente
   - `git clone`

2. Actualización de cambios  para traer a mi maquina lo que esta en GitHub debo de hacer un pull
   - `git pull`

3. Comandos para hacer y examinar los cambios en nuestro proyecto 
   - `git add`
   - `git commit`

4. Fusion de los cambios  
   - Pull Request (merge)
   - tal como se expluca en el documento el hacer un pull request nos permite realizar una fusion del trabajo de una manera sencilla

5. Resolución de conflictos  
   - Edición manual del archivo

6. Elegir la primer version de lo que viene siendo esta practica
   - `git tag v1.0`


## Simular un conflicto de codigo y resolverlo

Se simuló un conflicto modificando el mismo archivo en dos branch diferentes 

La manera de resolverlo fue juntando cada una de las dos versiones en una misma (merge)

##  Conclusión y tareas resuletas 

Lo que realizo en esta practica fue:


Crear un repositorio en GitHub con la estructura de la arquitectura empresarial.
Realizar al menos tres "commits" que representen el ciclo de vida del desarrollo.
Simular un conflicto de código y resolverlo mediante una fusión (merge), demostrando el dominio del desarrollo colaborativo.

El usar estos controladores de versiones no solo nos facilta en cierto aspecto el trabajo colecitvo, de igual manera nos permite tener un control de nuestro proyecto a 
la hora de querer escalarlo o de igual manera a la hora de equivocarnos se nos permitiria corregirlo.
Son herramientas a las cuales le podemos sacar mucho provecho.
